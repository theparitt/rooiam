## Phase 4 — Protocol Maturity + Security Hardening

*Goal: push Protocol to 10/10 and Security to 10/10.*
*These are the highest-leverage items because they make the existing platform trustworthy and interoperable.*

### Protocol Correctness (9 → 10)

#### Standards compliance
- [ ] RFC-grade negative path coverage — invalid redirect URI, bad PKCE verifier, reused auth code, reused rotated refresh token, expired refresh token, bad scope, nonce mismatch
- [ ] Exact RFC-style error response shapes (`error`, `error_description`, `error_uri`) on all OAuth/OIDC endpoints
- [ ] Stricter client authentication — confidential clients must always send `client_secret`; public clients must never send one
- [ ] Scope handling hardening — unknown scopes rejected, scope downgrade on refresh enforced
- [ ] Stable claim guarantees — `sub` is permanent; `email` and `name` only in token when scoped correctly

#### Logout strategy
- [ ] `end_session_endpoint` — RP-initiated logout (clears session, redirects to `post_logout_redirect_uri`)
- [ ] Front-channel logout support (iframe-based, optional per client)
- [ ] Back-channel logout support (server-to-server POST, optional per client)
- [ ] `id_token_hint` validation on logout

#### Interoperability
- [ ] Verified integration with NextAuth / Auth.js
- [ ] Verified integration with oauth2-proxy
- [ ] Verified integration with a plain SPA PKCE client
- [ ] Verified integration with a mobile OIDC library (AppAuth)
- [ ] httpYac negative-path test suite (extend `/test/` coverage)

### Security Architecture (8.5 → 10)

#### Key lifecycle
- [ ] OIDC signing key rotation — new key generated on demand; old key stays in JWKS for `kid` lookup until all issued tokens expire
- [ ] Multiple active keys in JWKS during rollover window
- [ ] `kid` claim in all issued JWTs matched to active key
- [ ] Emergency key rollover runbook documented

#### Identity lifecycle safety
- [ ] Secure email change flow — verify new address before switching; old address notified
- [ ] Account recovery rules — what happens when all auth methods are lost
- [ ] Provider unlink safety — block unlink if it would remove last login method
- [ ] Anti-lockout safeguards — owner cannot remove last owner from workspace

#### Session security
- [ ] **Device trust** — `trusted_devices` table; post-MFA device tagging; trusted devices skip MFA prompt; My Account → Trusted Devices tab to review and revoke
- [ ] Step-up auth (`acr_values`) — re-authenticate before sensitive operations (email change, passkey removal, owner transfer)
- [ ] Anomaly scoring — combine session binding mismatch + suspicious login signals into a risk score; higher-risk sessions get shorter TTL

#### Security operations
- [ ] Admin-visible suspicious activity queue — platform admin can see recent `auth.login.suspicious` and `auth.session.binding_mismatch` events in one place
- [ ] Alert thresholds — configurable: N failed logins in T minutes → notify platform admin

## Phase 5 — Control Plane + Developer Ecosystem

*Goal: push Control plane to 10/10 and Developer ecosystem to 8/10.*
*This is Rooiam's differentiation and the primary adoption driver.*

### Control Plane Maturity (8 → 10)

#### Policy clarity
- [ ] Effective policy view — for every policy dimension (IP, login methods, MFA, session), show: platform default → tenant override → workspace override → **effective result**
- [ ] Policy change preview — show what will break before saving an auth policy change (e.g. "disabling magic link will affect 3 users with no other login method")
- [ ] Safe rollback — last-N auth policy snapshots stored; one-click restore from audit log

#### Permission system UX
- [ ] Permission catalog page — list all 9 permissions with plain-language descriptions
- [ ] Role diff view — compare two roles side by side
- [ ] Built-in role templates — "Billing Admin", "Support Agent", "Auditor" as one-click starting points for custom roles
- [ ] "Why access denied?" — when a handler returns 403, include which permission was missing

#### Delegation
- [ ] Scoped admin delegation — owner can grant another member temporary elevated access with an expiry
- [ ] Approval model (optional) — sensitive policy changes require a second owner to confirm

#### Operational visibility
- [ ] Active sessions by workspace — platform admin sees current active session count per workspace
- [ ] Auth method usage stats — how many users in a workspace use each login method (informs safe policy changes)
- [ ] Tenant health overview — workspace-level: member count, active sessions, last login, policy flags

#### Admin safety
- [ ] Self-lockout prevention — block auth policy change that would make the acting admin unable to log in
- [ ] Owner transfer flow — explicit handoff with confirmation, not just role reassignment

### Developer Ecosystem (4 → 8)

#### JavaScript / TypeScript SDK
- [ ] `@rooiam/client` package — login redirect helpers, token verification, user extraction, logout
- [ ] Next.js integration guide + middleware helper
- [ ] SPA PKCE reference app (Vite + React)
- [ ] Express / Node backend token validation guide

#### Rust SDK
- [ ] `rooiam` crate — token verification, middleware for Actix Web, user extraction
- [ ] Actix Web integration guide

#### Developer docs
- [ ] Public docs site (extend `rooiam-docs`) — quickstart-first, use-case structured
- [ ] Quickstart: working login in under 15 minutes
- [ ] Client creation guide (web, SPA, native, machine-to-machine)
- [ ] Token verification guide (with `kid` rotation awareness)
- [ ] Error reference — every `error` code with human explanation

#### Dev tooling
- [ ] Docker Compose quickstart — single `docker compose up` gives a working Rooiam instance with demo seed
- [ ] Environment template (`.env.example`) with every variable documented
- [ ] Versioned changelog — every release lists breaking changes, claim changes, migration notes

## Phase 7 — Enterprise Federation

*Goal: push Enterprise federation from 2/10 to 10/10. Only after real demand.*

### SAML
- [ ] SAML 2.0 SP — accept assertions from enterprise IdPs
- [ ] SP metadata generation and consumption
- [ ] Assertion validation (signature, audience, NotBefore/NotOnOrAfter)
- [ ] Certificate management — upload IdP cert, rotate SP cert
- [ ] Attribute mapping — map SAML attributes to Rooiam user fields
- [ ] IdP-specific compatibility (Okta, Azure AD, Google Workspace)

### SCIM
- [ ] User provisioning — create/update user from SCIM `POST /Users`
- [ ] Deprovisioning — disable user on SCIM `DELETE /Users/{id}`
- [ ] Group sync — map SCIM groups to workspace roles
- [ ] PATCH semantics — partial updates per RFC 7644
- [ ] External directory identifiers — store `externalId` on users

### Directory integration
- [ ] LDAP / Active Directory read sync
- [ ] JIT provisioning from SAML / SCIM — create user on first successful assertion
- [ ] Directory group → workspace role mapping
- [ ] Staged rollout — enable federation per workspace, not platform-wide

### Enterprise compliance
- [ ] GDPR right to erasure — full user data deletion cascade
- [ ] PII masking in audit logs (configurable per platform)
- [ ] SOC2 / ISO 27001 audit report export
- [ ] Data residency controls

### Enterprise admin workflows
- [ ] Federation connection setup UX — guided wizard per provider
- [ ] Federation troubleshooting log — show last N assertion attempts with error detail
- [ ] Attribute mapping preview — dry-run before enabling
- [ ] Impersonation — platform admin can act as a user with full audit trail
