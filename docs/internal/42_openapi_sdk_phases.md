# OpenAPI + SDK ‚Äî Phased Plan

Status: **In progress ‚Äî Phases A‚ÄìD done + 3 D-backfills; browser + OIDC surface 100% covered (2026-06-02). Spec at 76 ops. Next: Phase E (refactor consumers).**

Goal: a typed, language-agnostic SDK layer so downstream apps integrate with
Rooiam through generated clients, not hand-rolled `fetch`. The server's OpenAPI
spec (generated by `utoipa`, so it cannot drift) is the single source of truth.

Background design: [41_sdk_plan.md](./41_sdk_plan.md) (two TS packages, monorepo,
generate-from-code). This doc is the **execution plan** with phases.

Hard rule (locked): **the SDK must be 100% stable + tested against the live
server before any working frontend (rooiam-admin, rooiam-app) is refactored onto
it.** Prove the layer first; move the apps last.

### The annotation recipe (mechanical ‚Äî repeat per endpoint)

The 30 remaining endpoints are pure repetition of three proven shapes. For each:

1. Make the handler **`pub`** (utoipa's generated `__path_*` item must be
   reachable from `openapi.rs`).
2. Add the `#[utoipa::path(...)]` attribute above the handler.
3. Register the handler in `src/openapi.rs` under `paths(...)`.
4. (If it has a named response struct, derive `utoipa::ToSchema` on it and add it
   to `components(schemas(...))`.)

**Shape 1 ‚Äî no params** (e.g. `workspace`, `roles`, `permissions`):
```rust
#[utoipa::path(get, path = "/v1/orgs/integrations/roles", tag = "integrations",
    security(("workspace_api_key" = [])),
    responses((status = 200, description = "..."), (status = 401), (status = 403)))]
```

**Shape 2 ‚Äî query params** (e.g. `members`, `activity`, `clients` list): add
`#[derive(utoipa::IntoParams)]` + `#[into_params(parameter_in = Query)]` to the
query struct, then:
```rust
#[utoipa::path(get, path = "...", tag = "integrations",
    params(SomeListQuery), security(("workspace_api_key" = [])),
    responses((status = 200, description = "...")))]
```

**Shape 3 ‚Äî path param** (e.g. `members/{member_id}`):
```rust
#[utoipa::path(get, path = "/v1/orgs/integrations/members/{member_id}", tag = "integrations",
    params(("member_id" = Uuid, Path, description = "Member UUID")),
    security(("workspace_api_key" = [])),
    responses((status = 200), (status = 404)))]
```

**Body shape ‚Äî POST/PATCH** (e.g. create client, update role): derive
`utoipa::ToSchema` on the request body struct, add `request_body = TheStruct`:
```rust
#[utoipa::path(patch, path = ".../{member_id}/role", tag = "integrations",
    params(("member_id" = Uuid, Path)), request_body = UpdateMemberRoleRequest, ...)]
```

Response schemas can stay loose (just `status` + `description`) initially and be
tightened to `body = SomeResponse` per endpoint later ‚Äî the path/params/auth are
what the SDK needs first.

Exit test: `/openapi.json` lists every `/orgs/integrations/*` path with correct
params (respecting `deny_unknown_fields`, `page_size` ‚â§ 1000); Swagger UI renders
them. Dump the spec without booting the DB via a throwaway example:
`examples/dump_openapi.rs` ‚Üí `ApiDoc::openapi().to_pretty_json()`.

**Build note:** generated types must live in `schema.ts` (NOT `schema.d.ts`) ‚Äî
`tsc` treats `.d.ts` as ambient and won't emit it to `dist`, which would break
`export type { paths } from './generated/schema.js'` for consumers. The `gen`
script outputs `schema.ts`.

stable + tested against the real server ‚Äî the bar for refactoring consumers is met.

## Phase D ‚Äî `@rooiam/sdk-browser` (TypeScript) ‚úÖ (core slice done 2026-06-02)

Cookie-based, NO secrets. Every request uses `credentials: 'include'`; auth is the
opaque `rooiam_session` cookie. A new `browser` tag + `session_cookie` (apiKey in
cookie) security scheme were added to the spec; `workspace_api_key` and
`session_cookie` now coexist. Spec is up to **40 operations** total.

Server endpoints annotated (7, all `pub`):
- [x] `GET /setup/auth-methods`, `GET /setup/login-bootstrap` (public; `PublicAuthMethodsQuery` got `IntoParams`)
- [x] `POST /auth/magic-link/start`, `POST /auth/magic-link/verify` (public; sets session cookie)
- [x] `POST /auth/logout`, `GET /identity/me`, `PATCH /identity/me/profile` (session cookie)
- DTOs deriving `ToSchema`: `StartMagicLinkRequest`, `VerifyMagicLinkRequest`, `UpdateProfileRequest`

SDK (`@rooiam/sdk-browser`, `RooiamBrowser`):
- [x] Public flow: `authMethods()`, `loginBootstrap()`, `startLogin()`, `verifyLogin()`
- [x] Session self-service: `me()`, `updateProfile()`, `logout()`
- [x] `RooiamError` (status + body); `credentials: 'include'` baked in; no Authorization header ever
- [x] Follows the widget redirect contract ([[project_rooiam_widget_redirect_contract]])

Exit test (stability bar) ‚Äî **CLEARED**: live-tested against the source-run server.
**16/16 pass** (13 unit + 3 live). Live block exercises the public endpoints
(`auth-methods`/`login-bootstrap` ‚Üí 200, `me()` ‚Üí 401 with no cookie). Session-cookie
calls aren't live-tested headlessly (would need a minted login cookie) ‚Äî covered by unit tests.

Deferred to a later Phase-D pass (not in the core slice): OIDC authorize/token/userinfo
+ PKCE helpers, webauthn/passkeys, mfa/totp, the rest of `/identity/me/*`
(sessions, linked-accounts, email-change, delete, audit-logs). The pipeline +
cookie-auth + the stability bar are proven; these are mechanical backfill.

### Phase D backfill #1 ‚Äî OIDC client flow + PKCE ‚úÖ (done 2026-06-02)

The standard OIDC login flow downstream apps use. New `oidc` tag +
`oidc_access_token` bearer scheme. Spec is up to **45 operations**.

Server endpoints annotated (5, already `pub`):
- [x] `GET /.well-known/openid-configuration` (discovery), `GET /.well-known/jwks.json`
      ‚Äî these live at the ORIGIN root, **not** under `/v1`
- [x] `GET /v1/oidc/authorize` (`AuthorizeRequest` ‚Üí `IntoParams`; 302 redirect)
- [x] `POST /v1/oidc/token` (`TokenRequest` ‚Üí `ToSchema`; **form-encoded**, so
      `request_body(content = TokenRequest, content_type = "application/x-www-form-urlencoded")`)
- [x] `GET /v1/oidc/userinfo` (`oidc_access_token` bearer)

SDK additions (`@rooiam/sdk-browser`):
- [x] PKCE helpers (Web Crypto, exported standalone): `generateCodeVerifier()`,
      `deriveCodeChallenge()`, `createPkcePair()` ‚Äî S256, verified against the
      RFC 7636 Appendix B test vector
- [x] `oidc.{discovery, jwks, authorizeUrl, exchangeCode, userinfo}` ‚Äî
      `authorizeUrl()` is pure string-building; `exchangeCode()` sends
      `application/x-www-form-urlencoded` with the verifier and NO client secret
      (public client); `userinfo()` sends the access token as a Bearer header

**Live contract finding #2:** discovery + JWKS are served at the **origin root**
(`/.well-known/*`), and `/v1/.well-known/*` is a 404. The SDK's normal `request()`
prepends `apiBase` (which includes `/v1`), so those two methods pass an explicit
`base: this.origin` override. Caught by the live test before shipping.

Exit test ‚Äî **CLEARED**: **25/25 pass** (20 unit + 5 live). Live block exercises
discovery + jwks against the source-run server; a manual probe confirmed
`POST /oidc/token` parses the form body (returns an OAuth error, not a 415).
