## What Rooiam is

Rooiam is an open-source, **self-hosted, passwordless IAM platform for
multi-tenant SaaS** (an OIDC provider + workspace admin system). It is licensed
Apache-2.0, currently at version **0.1**.

**Stack:**
- `rooiam-server` — Rust, Actix-web 4, PostgreSQL (via sqlx), Redis, MinIO
  (object storage), Mailhog (dev email). Serves a REST API under `/v1/...` and
  OIDC endpoints.
- `rooiam-admin` — React + Vite + TypeScript. Platform **operator** console
  (port 5171). For the person running the whole instance.
- `rooiam-app` — React + Vite + TypeScript. **Tenant** portal + hosted login
  widget (port 5172). For the businesses (workspaces) using the instance.
- `candycloud-web` / `candycloud-server` — a downstream **end-user** demo app
  that logs in via Rooiam (the "customer of a tenant" surface).
- `rooiam-examples` — integration examples.

**Auth methods today:** magic link, passkey (WebAuthn), Google OAuth, Microsoft
OAuth. No username/password (passwordless by design). OIDC + PKCE for downstream
apps. Opaque session cookies. Workspace API keys (Argon2id-hashed) for
server-to-server integration.

**Key server conventions (important — these caused real bugs before):**
- Request DTOs use `#[serde(deny_unknown_fields)]` — sending an unknown field =
  HTTP 400. Clients must send ONLY the params an endpoint accepts.
- `page_size` is capped (currently 1000) — oversized = 400.
- Strict env contract: wrong-mode env vars fail startup; the server validates
  config and exits on mismatch.
- The server resolves login redirect targets server-side from the DB; clients
  send `workspace_id` + `client_id`, not redirect URLs.
- No silent fallbacks for routing/identity values — missing = explicit error.

**Deployment:** Docker Compose. Two axes — mode (`demo` | `production`) chosen by
the compose file, target (`local` | `public`) chosen by the `.env.docker.*` file.
Postgres/Redis/MinIO/Mailhog are bundled in the compose stack.

## Goal of v0.2

Make Rooiam **easy to integrate** (a typed SDK over a generated API contract) and
**easy to run** (clear self-host config). This is the foundation the later
roadmap (device/QR login) depends on — a clean, stable API contract first.

There are two parallel tracks.

---

## Track 1 — OpenAPI + SDK (the integration foundation)

**Why:** downstream apps currently hand-roll `fetch` calls against the API, which
caused contract bugs (oversized page_size, unknown-field rejections, wrong
redirect params). A typed SDK generated from the server's own OpenAPI contract
makes those bugs impossible.

**Approach:** the server generates its OpenAPI spec **from the Rust code** using
the `utoipa` crate (so the spec can never drift from the code). The TypeScript
SDKs are then generated/derived from that spec.

### Phase A — OpenAPI foundation on the server  ✅ DONE
- Added `utoipa` + `utoipa-swagger-ui` to the server.
- Created an `ApiDoc` aggregator (`src/openapi.rs`).
- Annotated the first endpoint (`GET /v1/orgs/integrations/workspace`) with
  `#[utoipa::path]` + `#[derive(ToSchema)]` on its response.
- Server serves the spec at `/openapi.json` and Swagger UI at `/docs`.
- Verified the generated spec is valid OpenAPI 3.1.0.

### Phase B — Annotate the integration surface  ⬜ NEXT
Annotate all `/v1/orgs/integrations/*` endpoints (~24) — the workspace-API-key
surface the server SDK wraps. Per endpoint: add `#[utoipa::path(...)]` to the
handler + `#[derive(ToSchema)]` to its request/response DTOs + register it in the
`ApiDoc` `paths(...)` / `components(schemas(...))`. Priority: members (list,
detail, sessions, activity, role, profile), activity (audit log), clients,
invites, then branding/auth-config/policy/roles/permissions/etc.
Exit test: `/openapi.json` lists every integration path with correct params and
response schemas; Swagger UI renders them.

### Phase C — `@rooiam/sdk-server` (TypeScript)  ⬜
Server-side SDK (uses a workspace **API key** — secret, never in a browser).
- Scaffold monorepo `rooiam-sdk/packages/js-server`.
- Generate base types from `/openapi.json` (e.g. `openapi-typescript`); add a
  thin runtime client.
- Hand-write ergonomic wrappers: `rooiam.members.list()`,
  `rooiam.activity.list()`, `rooiam.clients.*`, plus the OIDC server bits
  (`exchangeCode`, `userinfo`).
- Exit/stability bar: a test suite hits a **live rooiam-server** with a real API
  key and exercises every method — all green.

### Phase D — `@rooiam/sdk-browser` (TypeScript)  ⬜
Frontend SDK (user session / OIDC — NO secrets).
- Scaffold `rooiam-sdk/packages/js-browser`.
- Mount the hosted login widget, OIDC + PKCE login/callback, getUser, logout,
  read public branding/auth-methods.
- Same stability bar (tested against live server).

### Phase E — Refactor consumers onto the proven SDK  ⬜
Only after C and D pass their tests. One at a time, each verified:
candycloud → rooiam-admin → rooiam-app (replace their hand-rolled fetch code).

**HARD RULE:** the SDK must be 100% stable + tested against the live server
BEFORE any working frontend is refactored onto it. Prove the layer first; move
the apps last.

**Packaging decision:** two separate TS packages — `@rooiam/sdk-browser` and
`@rooiam/sdk-server` — NOT one package. The browser package must never contain
API-key code. Monorepo organized by language; only JS splits into browser+server
(every other language is server-only, one package each). Adopt
`openapi-generator` for non-TS languages later when a second language is needed.

## Track 2 — Self-host trust + integration clarity

- deployment doctrine clarity (one clear Docker guide)
- explicit public-URL and mode behavior
- operator install flow
- production setup docs
- backup / restore guidance
- upgrade guidance
- one gold-standard hosted-login integration path
- one gold-standard OIDC app integration path

**Success test for v0.2:** a technical team can self-host Rooiam and complete a
first integration using the SDK + docs without reading much server source code.

---

## What is OUT of scope for v0.2

- Device / QR phone login (that's v0.3+, a separate plan — it depends on this
  foundation being done first).
- Username/password login (contradicts the passwordless positioning; deferred,
  opt-in only if ever).
- SAML, SCIM, multi-region, enterprise breadth (post-v1.0).
